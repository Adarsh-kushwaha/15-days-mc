React Rendering Internals
React Core
▼
What you must deeply understand today
  • React Fiber: Why it replaced the stack reconciler. Fiber = unit of work. React can pause, prioritize, resume work — this is WHY concurrent mode exists.
  • Reconciliation: React diffs the new VDOM against old. Same type = update props. Different type = unmount + remount. Key prop tells React "same element, different position" — why key=index breaks animations/forms.
  • Render vs Commit phase: Render is pure (calculate changes). Commit is where DOM mutations happen. Effects run AFTER commit.
  • When does a component re-render? State change, prop change, parent re-render, context value change. React.memo only prevents prop-change re-renders.
  • React 18 — Automatic batching: Before React 18, state updates in setTimeout/promises were NOT batched. Now they are. This changes behavior — know it.
🔬 Verify your understanding
  • Can you predict which components re-render in a nested tree when parent state changes?
  • Can you explain why key=index breaks a list with deletions?
  • Can you describe what happens between clicking a button and seeing a DOM update?

useState + useEffect Deep Internals
React Hooks
▼
  • useState closure trap: Event handlers capture state at the time they were created. setState with stale state is the #1 React bug. Fix: functional updates setState(prev => prev + 1).
  • useEffect dependency array: Empty [] = run once after mount. [dep] = run when dep changes. No array = run after every render. Return function = cleanup before next run OR unmount.
  • useEffect vs useLayoutEffect: useEffect fires after paint (async). useLayoutEffect fires before paint (sync). Use layout effect when you need to measure DOM before user sees it (no flicker).
  • React 18 — useEffect fires twice in Strict Mode (dev only): React unmounts and remounts to find cleanup bugs. Your cleanup MUST be correct or you get double API calls, double subscriptions.
⚡ Implement these from scratch

  • Build a timer that correctly cleans up on unmount
  • Show useLayoutEffect preventing a visible flicker (measure + reposition)

useMemo, useCallback, React.memo — When They Actually Help
Performance
▼
  • React.memo: Wraps a component. Skips re-render if props are shallowly equal. PROBLEM: objects/arrays/functions are new references every render. memo does nothing if parent passes {} or [] inline.
  • useCallback: Returns the same function reference between renders. Useful ONLY when that function is passed as a prop to a memo'd child. Otherwise it's just overhead.
  • useMemo: Memoizes an expensive computation. Only worth it if calculation takes measurable time. Don't wrap simple math — you pay the memo cost for nothing.
  • The golden rule: Profile first. Add memoization only when React DevTools Profiler shows unnecessary re-renders causing jank. Premature memo = worse performance, harder-to-read code.
  • useRef: Persists a value across renders WITHOUT causing re-render. Ideal for: previous value tracking, DOM references, storing interval/timeout IDs, tracking if component is mounted.
⚡ Prove your understanding
  • Build a child that logs when it renders. Show it still renders even with React.memo when parent passes inline function — then fix with useCallback.
  • Build a usePrevious hook using useRef

useContext + useReducer + React 18 Features
React Advanced
▼
  • useContext performance trap: Every consumer re-renders when context value changes — even if the consuming component only uses part of the value. Fix: split contexts, or wrap value in useMemo.
  • useReducer: For complex state with multiple sub-values or when next state depends on previous. Think Redux-style but local. Action pattern makes state transitions explicit and testable.
  • startTransition (React 18): Marks an update as non-urgent. React can interrupt it to handle user input. Use for: search filtering, tab switching, anything where immediate feedback matters more than the update completing.
  • useDeferredValue (React 18): Like a debounce but smarter — defers a value update until React is idle. Use for expensive derived renders (filtered lists, search results).
  • Suspense for data fetching: Component can "suspend" while waiting for data. Works natively with React Query, SWR, and Next.js. Know the concept even if you don't use it daily.

React Patterns: Compound Components + HOC + Render Props
Patterns
▼
  • Compound Components: Parent + children that implicitly share state via Context. Example: <Select> + <Option>. The component API is declarative and flexible. This is what all great UI libraries do.
  • HOC (Higher Order Component): Function that takes a component and returns enhanced component. withAuth, withLogger. Problem: prop collision, wrapper hell. Largely replaced by hooks but STILL asked in interviews.
  • Render Props: Component accepts a function as children or prop. Flexible inversion of control. Also largely replaced by hooks but know it for interviews and understanding libraries like react-table.
  • When to use what: Hooks for logic reuse. Compound components for flexible UI API. HOC for cross-cutting concerns that must wrap (auth, analytics). Know the tradeoffs cold.

Error Boundaries + Code Splitting + State Architecture
React Advanced
▼
  • Error Boundaries: Class component with componentDidCatch + getDerivedStateFromError. Catches render errors below them. Doesn't catch: async errors, event handlers, server errors. Place strategically — feature-level, not component-level.
  • React.lazy + Suspense: Dynamic import wraps a component. Suspense shows fallback while loading. Use for route-level splitting. Heavy components (rich text editor, charts).
  • State colocation: Keep state as close to where it's used as possible. Don't lift state to global if only one subtree needs it. This prevents unnecessary re-renders across the app.
  • Zustand vs Redux Toolkit vs Context: Context = simple sharing, no actions. Zustand = simple global state with minimal boilerplate. Redux Toolkit = complex state, time-travel debugging, large teams. Know the tradeoffs, not just the APIs.

10:00 – 12:00
Real-time, Performance, Security — SD Follow-up Topics
System Design
▼
  • WebSocket vs SSE vs Polling: WebSocket = full-duplex (chat). SSE = server→client only, simpler, auto-reconnect (live feeds, notifications). Polling = simple but wasteful. Know which to suggest and why.
  • Core Web Vitals: LCP (largest content paint, target <2.5s) — fix with preloading, image optimization, fast server. FID/INP (input delay) — fix with code splitting, avoiding long tasks. CLS (layout shift) — fix with explicit dimensions, font loading.
  • SSR vs CSR vs SSG vs ISR: CSR = React default, bad for SEO, fast after load. SSR = HTML from server per request, good for SEO + dynamic. SSG = pre-rendered at build time, fastest. ISR = SSG + background revalidation. Use ISR for blog, SSR for dashboard, CSR for admin tools.
  • XSS: Reflected (URL), Stored (DB), DOM-based. Prevention: sanitize inputs, escape output, Content Security Policy header, React's JSX auto-escapes (don't use dangerouslySetInnerHTML with user input).
  • CSRF: Attacker tricks browser into sending authenticated request. Fix: SameSite=Strict cookies, CSRF tokens, check Origin header.

Virtual List / Windowing — Theory First
Performance
▼
